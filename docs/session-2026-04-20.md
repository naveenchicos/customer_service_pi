## Issues Encountered and Resolutions

### 1. `docker-credential-gcr: executable file not found in $PATH`

**Symptom:** `docker build` failed immediately with:
```
error getting credentials - err: exec: "docker-credential-gcr": executable file not found in $PATH
```

**Root cause:** `~/.docker/config.json` had a stale entry from a previous Docker Hub login:
```json
"https://index.docker.io/v1/": "gcr"
```
This told Docker to use `docker-credential-gcr` as the credential helper for Docker Hub, but that binary is not installed.

**Fix:** Changed the entry to use the `desktop` credential store instead:
```json
"https://index.docker.io/v1/": "desktop"
```
File: `~/.docker/config.json`, line 13.

---

### 2. GCE quota prevented multi-zone node provisioning

**Symptom:** `kubectl rollout status` timed out. Pod events showed:
```
Warning  FailedScaleUp  Node scale up in zones us-central1-b associated with this pod failed: GCE quota exceeded.
```

**Root cause:** The `topologySpreadConstraints` in `deployment.yaml` requires pods to be spread across availability zones (`topology.kubernetes.io/zone`). GKE Autopilot tried to provision one node per zone (b, c, f) but the GCP project had insufficient CPU quota to provision 3 nodes simultaneously.

**What happened:** GKE eventually scheduled all 3 pods on a single node (`gk3-ol-cluster-nap-12iqsozi-4f683a4b-tn8m` in `us-central1-b`) after quota for the other zones failed. Pods still started but without zone redundancy.

**Fix applied:** None — pods ran successfully on the single node. The `topologySpreadConstraints` with `whenUnsatisfiable: DoNotSchedule` was relaxed by GKE Autopilot's scheduler under quota pressure.

**Permanent fix (TODO):** Request a quota increase in GCP Console → IAM & Admin → Quotas → filter for `CPUS` or `N2_CPUS` in `us-central1`. Request at least 6 CPUs across 3 zones to support multi-zone pod spreading.

---

### 3. Pods took ~8 minutes to start (slow liveness probe initially)

**Symptom:** Pods showed `1/2 Running` (app container running, proxy container starting), then liveness probe failed with `connection refused` on port 8000.

**Root cause (investigation):**
- Cloud SQL Auth Proxy image (`gcr.io/cloud-sql-connectors/cloud-sql-proxy:2.14.0`) is 16MB and pulled slowly from gcr.io on a fresh node
- The app container started first (at 22:56:06) but the Cloud SQL proxy didn't finish pulling until 22:57:15 — a 69-second gap
- The app was in the uvicorn startup phase during this time (epoll_wait, asyncio event loop running, no port 8000 yet)
- After one restart, all images were cached on the node and pods came up quickly

**Diagnostic commands used:**
```bash
cat /proc/net/tcp          # confirmed port 8000 not yet listening (then later confirmed LISTEN)
cat /proc/1/wchan          # do_epoll_wait — asyncio event loop running normally
cat /proc/1/status         # State: S (sleeping) — process alive
ls /proc/1/fd/             # 3 open sockets confirmed uvicorn had bound the port
```

**Resolution:** Pods auto-recovered after one restart. All 3 are now `2/2 Running` with health probes passing.
