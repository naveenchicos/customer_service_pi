### 1. Cloud SQL Auth Proxy 403 — node compute SA has no Cloud SQL access

**Symptom:**
```
ERROR: failed to get instance metadata: googleapi: Error 403: boss::NOT_AUTHORIZED:
  Not authorized to access resource. Possibly missing permission cloudsql.instances.get
```

**Root cause:** We removed `serviceAccountName` from the deployment (Workload Identity deferred).
The proxy fell back to Application Default Credentials → GKE node compute service account
(`393613539580-compute@developer.gserviceaccount.com`) → no Cloud SQL permissions.

**Fix:** Mounted SA key as K8s Secret and set `GOOGLE_APPLICATION_CREDENTIALS` on the proxy container.
See `k8s/deployment.yaml` `cloud-sql-sa-key` volume.

**Future fix:** Migrate to Workload Identity Federation — bind the K8s service account to
`pi-ai-dev-engineer@pi-dev-ai-493823.iam.gserviceaccount.com`. Requires project owner to run:
```bash
gcloud iam workload-identity-pools create "github-pool" \
  --project="pi-dev-ai-493823" --location="global"
```

### 2. `docker-credential-gcr: executable file not found in $PATH`

**Symptom:** `docker build` failed immediately — before pulling any image.

**Root cause:** `~/.docker/config.json` had a stale entry:

This told Docker to use `docker-credential-gcr` for Docker Hub credentials.
That binary is not installed locally.

**Fix:** Changed to:
```json
"https://index.docker.io/v1/": "desktop"
```

### 9. CD smoke test — `container not found` on exec

**Symptom:** CD smoke test step failed:
```
error: Internal error occurred: unable to upgrade connection:
  container not found ("ol-service")
```

**Root cause:** `kubectl exec` was targeting a pod from the old ReplicaSet that was in
`Terminating` → `Failed` phase immediately after rollout. The pod passed `condition=Ready`
briefly during the `kubectl wait` but was already being torn down.

**Fix:** Added `--sort-by=.metadata.creationTimestamp` and used `{.items[-1].metadata.name}`
to always select the newest Running pod (guaranteed to be from the new ReplicaSet):
```bash
kubectl wait pod -n production -l app=ol-service \
  --for=condition=Ready --timeout=60s
POD=$(kubectl get pod -n production -l app=ol-service \
  --field-selector=status.phase=Running \
  --sort-by=.metadata.creationTimestamp \
  -o jsonpath='{.items[-1].metadata.name}')
```

### Issues encountered and fixes

---

#### Issue 1 — `gcloud apigee envgroups` command does not exist

```
ERROR: (gcloud.apigee) Invalid choice: 'envgroups'
```

**Root cause:** The `gcloud apigee` CLI only covers a subset of Apigee operations.
`envgroups` is not exposed as a subcommand.

**Fix:** Use the Apigee Management REST API directly:
```bash
TOKEN=$(gcloud auth print-access-token)
curl -X PATCH \
  "https://apigee.googleapis.com/v1/organizations/pi-dev-ai-493823/envgroups/eval-group" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"hostnames": ["your-hostname"]}'
```
Same pattern applies to any Apigee resource the `gcloud` CLI doesn't expose.

---

#### Issue 2 — No real domain for Apigee hostname / SSL cert

**Root cause:** Apigee X requires a real hostname with DNS for the env group and managed SSL cert.

**Fix:** Use [nip.io](https://nip.io) — a free wildcard DNS service.
`api.34-107-235-25.nip.io` automatically resolves to `34.107.235.25`. No registration needed.
Google managed SSL certs work with nip.io because it provides real DNS resolution.

```bash
# Update env group
curl -X PATCH ".../envgroups/eval-group" -d '{"hostnames": ["api.34-107-235-25.nip.io"]}'

# Create SSL cert
gcloud compute ssl-certificates create apigee-ssl-cert-nip \
  --domains=api.34-107-235-25.nip.io --global
```

---

#### Issue 3 — TLS handshake fails (`SSL_ERROR_SYSCALL`) after SSL cert becomes ACTIVE

**Root cause:** The Global External LB backend service was forwarding requests to Apigee via PSC,
but Apigee didn't know which env group to route to because no `Host` header matched.

**Fix:** Set a custom request header on the backend service so Apigee can match the env group:

```bash
gcloud compute backend-services update apigee-backend \
  --global \
  --custom-request-header="Host: api.34-107-235-25.nip.io"
```

---

#### Issue 4 — `Invalid API call as no apiproduct match found` (401)

**Root cause:** API product had `apiResources: ["/accounts/**", "/health"]`.
Apigee's product path matching did not match the suffix `/accounts` against the wildcard `/**`.
Changed to `["/**", "/"]` but the cache hadn't cleared.

**Fix:** Set `apiResources: []` (empty array = all resources allowed, no path restriction).
This is the correct setting for a product that covers an entire proxy.

---

#### Issue 5 — Backend returns `Missing or invalid API key` instead of `Missing caller identity`

**Root cause:** After the middleware swap commit, the pod was still running the **old image**.
The CD pipeline uses `kubectl set image` with the `:main` floating tag. Since the tag string
didn't change between pushes, Kubernetes never triggered a rollout — pods kept running stale code.

**Fix (immediate):** `kubectl rollout restart deployment/ol-service -n production`

**Fix (permanent):** Changed CD workflow to derive a unique `sha-<7char>` tag per commit:

```yaml
- name: Set deploy tag
  id: deploy_tag
  run: echo "value=sha-$(echo '${{ github.sha }}' | cut -c1-7)" >> $GITHUB_OUTPUT
```

Now every push deploys a uniquely tagged image, guaranteeing a real rollout.
